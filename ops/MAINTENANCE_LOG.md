## 2026-07-17 (same day, later still) — Fixed false-positive `tekeche-oke-node-down` alarm

- **Type**: Bug fix, pre-existing alarm (not created this session) — MEDIUM per `Get-ChangeRisk.ps1`, but real blast radius near zero (monitoring-query-only, no functional/production resource touched)
- **Trigger**: user forwarded a real CRITICAL email — alarm fired at 2026-07-17T13:18 UTC claiming "fewer than 2 OKE nodes report ACTIVE condition." The alert's own payload was self-contradicting: both listed nodes (`...-2`, `...-3`) showed `nodeState=ACTIVE`/`nodeCondition=ACTIVE`, yet two *separate* metric streams each reported `count()=1.00`.
- **Root cause, confirmed via `oci monitoring metric list`**: OKE publishes `NodeState` as one distinct metric stream **per node per possible condition value** (ACTIVE/DELETING/UPDATING/etc.), not one pooled stream per cluster. The alarm's query (`NodeState[1m]{...}.count() < 2`, no `.grouping()`) therefore gets evaluated independently per node-stream — MQL's `.count()` counts *datapoints within a stream over the interval*, not *distinct streams/nodes*. Each healthy node's own ACTIVE-condition stream reports exactly 1 datapoint per minute (itself), which is always `< 2`, so the alarm fires essentially at random per node regardless of real cluster health. Confirmed ground truth via `oci compute instance list`: exactly 2 OKE node instances exist, both `RUNNING` — this was a false positive, not a real incident. Confirmed via `oci monitoring alarm-status list-alarms-status` that it had already self-resolved back to `OK` before this was even investigated (a transient flap, consistent with the bug).
- **Fix**: added `.grouping(clusterId)` to the query, collapsing all per-node ACTIVE streams into one series per cluster before `.count()` runs, so it now sums datapoints across every currently-ACTIVE node into one true total before comparing to 2.
- **Recovery point**: `2026-07-17_19-53-52_before-fix-false-positive-oke-node-down`.
- **Verification**: `terraform plan` confirmed isolated scope (`0 add, 1 change, 0 destroy` — only this alarm's `query` string). `oci monitoring alarm get` post-apply confirms the live query now includes `.grouping(clusterId)`. `Test-Build.ps1`: 8/9, same pre-existing unrelated gap, no regression.
